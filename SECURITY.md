## Reporting a Vulnerability

If you discover a security vulnerability, please report it responsibly.

### DO NOT

- Open a public GitHub issue for security vulnerabilities
- Post vulnerability details in discussions or comments
- Share exploits publicly before a fix is available

### DO

1. **Email**: Send details to <askb23@gmail.com>
2. **Subject**: `[SECURITY] zeroclaw-mission-control: <brief description>`
3. **Include**:
   - Description of the vulnerability
   - Steps to reproduce
   - Potential impact
   - Suggested fix (if any)

### Response Timeline

- **Acknowledgment**: Within 48 hours
- **Assessment**: Within 7 days
- **Fix/Mitigation**: Within 30 days (critical issues prioritized)

## Security Design Principles

This repository follows a zero-trust security model:

1. **No secrets in git** — All credentials stored in `.env` (gitignored)
2. **Token-based auth** — Gateway and dashboard use cryptographic tokens
3. **Network isolation** — Docker bridge network, internal-only Redis
4. **Non-root containers** — All services run as unprivileged users
5. **Read-only filesystems** — Where operationally feasible
6. **No privilege escalation** — `no-new-privileges` security option
7. **Resource limits** — CPU/memory caps prevent DoS
8. **Supply chain security** — All GHA actions SHA-pinned, Dependabot enabled
9. **Secret rotation** — `scripts/rotate-secrets.sh` for credential cycling
10. **Detection** — `detect-secrets` pre-commit hook, private key scanning

## Secret Management

### Initial Setup

```bash
# Generate all secrets automatically
./scripts/setup.sh
```

### Rotation

```bash
# Rotate all tokens and passwords
./scripts/rotate-secrets.sh
```

### What Goes in `.env`

```bash
# API provider keys
ANTHROPIC_API_KEY=
OPENAI_API_KEY=

# Internal tokens (auto-generated by setup.sh)
ZEROCLAW_GATEWAY_TOKEN=
REDIS_PASSWORD=
```

### What NEVER Goes in Git

- API keys (Anthropic, OpenAI, Brave Search)
- Gateway tokens
- Redis passwords
- TLS certificates/keys
- Any `.env` file
